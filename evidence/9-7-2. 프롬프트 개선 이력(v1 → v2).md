# 9-7-2. 프롬프트 개선 이력(v1 → v2)

본 항목은 시스템 프롬프트 v1을 실제 회의 메모에 적용한 결과를 바탕으로, v2 프롬프트에서 어떤 점을 개선했는지 정리한 내용입니다.
또한 기능 요구사항의 “단계적 추론 유도 적용 및 전/후 비교”를 반영하여, v1은 일반 지시 기반 프롬프트, v2는 단계적 접근을 유도하는 프롬프트로 구분해 비교했습니다.

---

## 1. 개선 목적

이번 개선의 목적은 같은 회의 메모 입력에 대해 일반 지시형 프롬프트와 단계적 접근 유도 프롬프트의 차이를 비교하고, 회의록 정리 결과의 안정성을 높이는 것입니다.

특히 본 과업에서는 단순히 문장을 보기 좋게 요약하는 것보다 다음 요소를 정확히 구분하는 것이 중요합니다.

* 결정사항
* 의견 또는 제안
* 확인 필요 사항
* 가능한 리스크
* 후속 실행 항목(Action Items)
* 확정되지 않은 기능 범위
* 담당 역할별 후속 업무

따라서 v2에서는 AI가 회의 메모를 바로 최종 회의록으로 작성하기보다, 내부적으로 먼저 내용을 분류하고 점검한 뒤 최종 결과를 출력하도록 개선했습니다.

---

## 2. v1 프롬프트의 특징

v1 프롬프트는 웹에이전시 프로젝트 담당자(PM)를 보조하는 회의 기록 및 프로젝트 문서화 담당자 역할을 AI에게 부여하고, B2B 제조기업 홈페이지 리뉴얼 킥오프 회의 메모를 사내 공유용 회의록으로 정리하도록 설계했습니다.

v1의 주요 특징은 다음과 같습니다.

* 회의 목적, 결정사항, 후속 실행 항목, 확인된 사실, 가능한 리스크, 확인 필요 사항, 후속 일정의 7개 항목으로 출력 구조를 고정했습니다.
* 회의 메모에 없는 내용을 임의로 작성하지 않도록 제한했습니다.
* 미확정 의견, 제안, 추정을 결정사항으로 작성하지 않도록 지시했습니다.
* 담당자, 기한, 선행 조건이 불명확한 업무는 확인 필요 사항으로 분리하도록 했습니다.
* 인물 실명 대신 역할명으로 작성하도록 했습니다.
* 사내 공유용 문서에 적합한 간결하고 명확한 문체를 사용하도록 했습니다.

v1은 기본적인 구조화 규칙과 안전장치를 포함하고 있지만, 회의 메모를 사실, 결정사항, 의견, 확인 필요 사항, 리스크 후보로 먼저 나누는 단계적 분류 절차는 상대적으로 약했습니다.

---

## 3. v1 테스트 조건

v1 프롬프트를 Claude Opus 4.8에 적용하여 기존 회의 메모를 사내 공유용 회의록으로 정리하도록 테스트했습니다.

테스트에 사용한 입력은 B2B 제조기업 홈페이지 리뉴얼 프로젝트 킥오프 회의 메모입니다. 해당 회의 메모에는 다음 내용이 포함되어 있습니다.

* 기존 홈페이지의 문제점
* 리뉴얼 1차 목표
* 홈페이지 주요 메뉴
* 제품소개 페이지 구성
* 고객문의 폼 구성
* 관리자 페이지 기능 요청
* 인증서 파일 관리 기능 포함 여부
* 게시판 데이터 이전 여부
* 다국어 페이지 필요 가능성
* 전체 오픈 목표일
* 담당 역할별 후속 업무

---

## 4. v1 실행 결과 요약

v1 실행 결과, Claude Opus 4.8은 전체적으로 안정적인 회의록 정리 결과를 생성했습니다.

잘 수행된 점은 다음과 같습니다.

* 요청한 7개 출력 항목을 모두 반영했습니다.
* 인증서 파일 관리 기능을 확정사항으로 단정하지 않고, 클라이언트 회신 대기 항목으로 분리했습니다.
* 게시판 데이터 이전, 다국어 페이지, 전체 오픈 목표일처럼 추가 확인이 필요한 항목을 가능한 리스크와 확인 필요 사항으로 구분했습니다.
* 사이트맵 확정 후 디자인, 디자인 확정 후 퍼블리싱이라는 업무 선후관계를 비교적 잘 보존했습니다.
* 회의 메모에 없는 구체적인 날짜나 범위를 임의로 만들어내는 문제는 발견되지 않았습니다.

다만 v2에서 개선할 수 있는 보완점도 확인되었습니다.

| 구분             | v1 결과에서 확인한 내용                                              | 개선 필요 이유                                                                                                    |
| -------------- | ----------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |

## 6. v2 개선 방향

v2 프롬프트는 v1의 기본 구조를 유지하되, 단계적 접근을 유도하는 방식으로 개선합니다.

v2의 핵심 개선 방향은 다음과 같습니다.

---

### 6-1. 결정사항 분류 기준 강화

v2에서는 “확정”, “포함하기로 함”, “전달하기로 함”처럼 명확한 합의 표현이 있는 경우에만 결정사항으로 분류합니다.

반면 다음과 같은 표현은 결정사항으로 단정하지 않습니다.

* 가능
* 가능할 것 같다
* 필요할 수도 있음
* 좋겠다
* 검토
* 회신 예정
* 내부 확인 필요

이러한 표현은 내용의 성격에 따라 확인된 사실, 가능한 리스크, 확인 필요 사항으로 분리합니다.

---

### 6-2. 후속 실행 항목 누락 방지 강화

v2에서는 최종 출력 전, 회의 메모에 등장한 주요 담당 역할이 후속 실행 항목 또는 확인 필요 사항에 반영되었는지 점검합니다.

명시된 기한이 없는 경우에도 조건부로 필요한 후속 검토 업무가 있다면 “기한 또는 조건”에 선행 조건을 표기합니다.

예를 들어 개발 담당의 업무가 명확한 일정으로 확정되지 않았더라도, 추가 기능 범위 확정 후 개발 범위·일정 재검토가 필요하다면 후속 실행 항목 또는 확인 필요 사항에 반영합니다.

---

### 6-3. 단계적 검토 방식 적용

v2에서는 AI가 바로 최종 회의록을 작성하기 전에, 내부적으로 다음 순서로 회의 메모를 검토하도록 합니다.

1. 회의 메모에서 확인된 사실을 먼저 분리합니다.
2. 확정된 결정사항과 의견·희망사항·검토 사항을 구분합니다.
3. 담당 역할별 후속 실행 항목을 정리합니다.
4. 기한, 담당자, 범위가 불명확한 항목을 확인 필요 사항으로 분리합니다.
5. 일정, 범위, 비용, 개발 난이도, 자료 전달 지연 가능성을 가능한 리스크로 분리합니다.
6. 최종 출력 전, 담당 역할 누락과 결정사항 오분류 여부를 점검합니다.

단, 이러한 단계적 검토 과정은 내부적으로만 수행합니다.

최종 출력에는 장문의 검토 과정이나 불필요한 추론 과정을 그대로 노출하지 않고, 회의록으로 바로 사용할 수 있는 정리 결과와 필요한 핵심 근거만 간단히 제시합니다.

---

### 6-4. 출력 용어와 형식 통일

v2에서는 최종 문서의 가독성을 위해 “Action Items” 대신 “후속 실행 항목”을 기본 항목명으로 사용합니다.

필요한 경우 최초 1회만 “후속 실행 항목(Action Items)”으로 병기하고, 이후에는 “후속 실행 항목”으로 통일합니다.

---

## 7. v1과 v2 비교표

| 구분        | v1: 일반 지시 기반 프롬프트                 | v2: 단계적 접근 유도 프롬프트                                           |
| --------- | --------------------------------- | ------------------------------------------------------------ |
| 기본 접근     | 역할, 출력 항목, 작성 규칙을 제시하고 회의록 정리를 요청 | 회의 메모를 내부적으로 사실, 결정사항, 의견, 확인 필요, 리스크 후보로 먼저 분류한 뒤 최종 회의록 작성 |
| 출력 항목     | 7개 항목 고정                          | 7개 항목 유지, “후속 실행 항목”으로 용어 통일                                 |
| 결정사항 분류   | 미확정 의견이나 제안을 결정사항으로 작성하지 않도록 제한   | “가능”, “검토”, “필요할 수도 있음”, “회신 예정” 표현은 결정사항에서 제외하도록 명시         |
| 후속 실행 항목  | 담당 역할, 작업, 기한 중심으로 정리             | 원문에 등장한 주요 담당 역할이 누락되지 않았는지 최종 점검                            |
| 기한 처리     | 기한이 없으면 확인 필요로 표시                 | 기한이 없더라도 선행 조건이 있으면 “기한 또는 조건”에 조건부로 표기                      |
| 환각 방지     | 회의 메모에 없는 내용 작성 금지                | 기한, 담당자, 기능 범위, 확정 여부를 임의 생성하지 않도록 자체 점검 강화                  |
| 단계적 추론 유도 | 기본 점검 질문은 있으나, 명시적인 단계적 분류 절차는 약함 | 사실 → 결정사항 → 의견·희망사항 → 확인 필요 → 리스크 후보 → 최종 출력 순서로 내부 검토       |
| 최종 답변 방식  | 구조화된 회의록 출력                       | 내부 검토 과정은 노출하지 않고, 회의록 정리 결과와 필요한 핵심 근거만 제시                  |

## 8. v2에서 추가할 핵심 규칙

v2 프롬프트에는 다음 규칙을 추가합니다.

* “가능”, “가능할 것 같다”, “필요할 수도 있음”, “좋겠다”, “검토”, “회신 예정”과 같은 표현은 확정된 결정사항으로 분류하지 않습니다.
* 회의 중 합의가 끝난 내용만 결정사항에 넣고, 개인 의견·제안·희망사항·클라이언트 회신 대기 사항은 확인 필요 사항 또는 가능한 리스크로 분리합니다.
* 최종 출력 전, 회의 메모에 등장한 주요 담당 역할이 후속 실행 항목 또는 확인 필요 사항에 반영되었는지 점검합니다.
* 명시된 기한이 없는 업무는 임의로 날짜를 만들지 않고, “확인 필요” 또는 선행 조건을 기준으로 표기합니다.
* 개발 범위, 추가 견적, 일정 변경 가능성이 있는 항목은 결정사항으로 단정하지 않고 가능한 리스크 또는 확인 필요 사항으로 분리합니다.
* 내부적으로는 회의 메모를 사실, 결정사항, 의견·희망사항, 확인 필요 사항, 가능한 리스크 후보로 단계적으로 검토합니다.
* 단, 최종 출력에는 내부 검토 과정이나 장문의 추론 과정을 그대로 노출하지 않습니다.
* 최종 답변에는 회의록으로 바로 사용할 수 있는 정리 결과와 필요한 경우 핵심 근거만 간단히 제시합니다.

---

## 9. 개선 결론

v1 프롬프트는 회의록 구조화, 리스크 분리, 확인 필요 사항 정리 측면에서 전반적으로 안정적인 결과를 보였습니다.

다만 실제 테스트 결과, “기본 범위에 포함 가능”처럼 확정과 가능성 사이에 있는 표현을 더 엄격하게 분류할 필요가 있었습니다. 또한 원문에 등장한 주요 담당 역할이 후속 실행 항목에서 누락되지 않도록 최종 점검 규칙을 강화할 필요도 확인했습니다.

따라서 v2에서는 결정사항 분류 기준, 후속 실행 항목 누락 방지, 단계적 검토 방식, 최종 출력 요약 규칙을 추가하여 회의록 요약 결과의 정확성과 실무 활용성을 높이는 방향으로 개선했습니다.
